# Resolving Merge Conflicts

Merge conflicts are **not errors**. They're a normal part of collaborative development. When two branches modify the same lines in the same file, Git cannot automatically decide which version to keep. It asks you—the human with domain knowledge—to make the call.

This unit teaches you to resolve conflicts confidently and cleanly.

## When Conflicts Occur

A conflict arises during a **three-way merge** when:

1. Two branches diverged (share a common ancestor but have different commits)
2. Both branches modified the **same lines** in the **same file**
3. Git cannot determine which version is correct

Example:

```
        E ← F  (feature: modified line 10)
       /
A ← B ← C ← D  (main: also modified line 10)
```

When merging, Git sees two different versions of line 10 and stops, asking you to decide.

**Fast-forward merges never conflict** because there's no divergence:

```
A ← B ← C ← D ← E ← F  (linear)
```

Here, `main` can simply move its pointer forward to F. No conflict.

## Anatomy of a Conflict

When a conflict occurs, Git marks the conflicted section in the file with three marker lines (shown as comments below to prevent Git from interpreting them):

```python
# <<<<<<< HEAD      # marks the start of HEAD (your) version
Your version (current branch)
# =======           # separates the two versions  
Their version (incoming branch)
# >>>>>>> feature-auth  # marks the end and shows incoming branch
```

**Example:**

Suppose `auth.js` has been edited on both `main` and `feature-auth`. Here's what a conflict looks like:

```javascript
// <<<<<<< HEAD
const TOKEN_EXPIRY = 3600; // 1 hour (version on main)
// =======
const TOKEN_EXPIRY = 7200; // 2 hours (version on feature-auth)
// >>>>>>> feature-auth
```

Git is saying:
- **HEAD** (your current branch, `main`) says: `3600`
- **feature-auth** (incoming) says: `7200`

You must choose which one (or write a new version that incorporates both).

### Multi-line Conflicts

Conflicts can span multiple lines:

```javascript
// <<<<<<< HEAD (version on main)
function authenticate(user, password) {
  validateInput(user, password);
  const token = generateJWT(user);
  return token;
}
// ======= (conflict separator)
// Version on feature-auth branch:
async function authenticate(user, password) {
  await validateInput(user, password);
  const hash = await hashPassword(password);
  const token = generateJWT(user, hash);
  return token;
}
// >>>>>>> feature-auth
```

You must resolve this by keeping, combining, or rewriting the code.

## Resolving Conflicts Step by Step

### 1. See What's Conflicted

When you attempt a merge and conflicts occur:

```bash
git merge feature-auth
```

Output:

```
Auto-merging auth.js
CONFLICT (content): Merge conflict in auth.js
Automatic merge failed; fix conflicts and then commit the result.
```

Check status:

```bash
git status
```

Output:

```
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  (use "git add <file>..." to mark as resolved)
        both modified: auth.js
```

### 2. Resolve the Conflict

Open the conflicted file in your editor and locate the markers (`<<<<<<<`, `=======`, `>>>>>>>`).

**Option A: Keep your version**

```javascript
const TOKEN_EXPIRY = 3600; // 1 hour
```

Delete the other version and the markers.

**Option B: Keep their version**

```javascript
const TOKEN_EXPIRY = 7200; // 2 hours
```

Delete your version and the markers.

**Option C: Combine versions**

```javascript
const TOKEN_EXPIRY = process.env.TOKEN_EXPIRY || 7200; // Default 2 hours
```

Write new code that incorporates elements from both.

### 3. Stage the Resolved File

Tell Git you've resolved the conflict:

```bash
git add auth.js
```

### 4. Complete the Merge

Finish the merge by creating the merge commit:

```bash
git commit -m "Merge branch 'feature-auth' with conflict resolution"
```

(Git pre-fills a merge commit message; you can accept or edit it.)

The merge is now complete. The conflicted file is integrated, and your branches are unified.

## Using Merge Helpers

### Git Status and Diff

See which files are conflicted:

```bash
git status
```

See the details of conflicts in one file:

```bash
git diff auth.js
```

Shows added/removed sections with markers.

### Merge Conflict Strategies

If you consistently want to favor one branch, use a strategy:

#### Take All of Ours (Your Version)

```bash
git merge feature-auth -X ours
```

All conflicts automatically resolved in favor of `HEAD` (your branch).

#### Take All of Theirs (Their Version)

```bash
git merge feature-auth -X theirs
```

All conflicts automatically resolved in favor of the incoming branch.

**Warning:** These strategies should be rare. Usually you need to actually decide, not auto-resolve.

### Abort a Conflicted Merge

If you start a merge and realize you're not ready to deal with conflicts:

```bash
git merge --abort
```

This cancels the merge entirely and returns to the pre-merge state.

## Practical Conflict Resolution

Let's walk through a real scenario:

```bash
# Working on main
git switch main
git add user-service.js
git commit -m "Add email field to user schema"

# Colleague pushed a conflicting change to main
git pull origin main
# Conflict!

# Check status
git status
# Both modified: user-service.js

# Open the file and resolve
# (Edit user-service.js, choose versions, remove markers)

# Stage the resolved file
git add user-service.js

# Complete the merge
git commit -m "Merge remote changes; resolved schema conflict"

# Push to remote
git push origin main
```

## Preventing Conflicts

While conflicts are normal, good practices reduce them:

1. **Commit frequently** with logical units — don't let branches diverge too long
2. **Communicate with teammates** — "I'm working on auth.js"
3. **Pull regularly** — `git pull` often to stay in sync
4. **Use feature branches** — isolate changes to a single feature
5. **Review PRs early** — catch conflicts before they're deeply nested

## Key Takeaways

- **Conflicts are not failures** — they're normal in collaborative development
- **Conflict markers** clearly show both versions; edit the file to resolve
- **Stage resolved files** to tell Git you've handled them
- **Complete merges** with a commit message
- **Prevention** is easier than resolution — communicate and sync regularly
