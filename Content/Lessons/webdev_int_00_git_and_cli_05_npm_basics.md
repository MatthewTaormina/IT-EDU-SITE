# npm Basics

> **Lesson Summary:** npm (Node Package Manager) is the tool that manages JavaScript dependencies for your project. It lets you install libraries written by others, lock their exact versions for reproducibility, and define scripts for common tasks like running a dev server or building for production.

---

## What Is Node.js?

**Node.js** is a JavaScript runtime that runs JavaScript outside the browser — on your machine, in a terminal, or on a server. It is required to use npm and most modern front-end build tools.

Check if it is installed:

```bash
node --version   # v20.x.x
npm --version    # 10.x.x
```

If not installed: download from [nodejs.org](https://nodejs.org/en/download) — choose the **LTS** (Long-Term Support) version.

> **💡 Tip:** Use **nvm** (Node Version Manager) instead of the direct installer if you will work with multiple projects that require different Node.js versions. Install nvm from [nvm.sh](https://nvm.sh).

---

## npm: The Package Manager

**npm** ships with Node.js. It serves three main purposes:

1. **Install packages** — download libraries from the npm registry
2. **Manage versions** — lock exact dependency versions for reproducibility
3. **Run scripts** — define and execute common project commands

The npm registry ([npmjs.com](https://www.npmjs.com)) hosts over 2 million packages.

---

## Initializing a Project — `npm init`

```bash
cd ~/projects/my-site
npm init -y
```

`npm init -y` creates a `package.json` file with default values (the `-y` flag skips the interactive questionnaire):

```json
{
  "name": "my-site",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}
```

---

## The `package.json` File

`package.json` is your project's manifest. The most important fields:

| Field | Purpose |
| :--- | :--- |
| `name` | The project name (used if you publish to npm) |
| `version` | The current version (semver: `major.minor.patch`) |
| `scripts` | Named shell commands you can run with `npm run <name>` |
| `dependencies` | Packages required in production |
| `devDependencies` | Packages required only during development (build tools, linters, test runners) |

---

## Installing Packages — `npm install`

```bash
npm install lodash              # install as a production dependency
npm install --save-dev vite     # install as a dev dependency
```

After installing, two things happen:
1. The package appears in `node_modules/`
2. The dependency is recorded in `package.json`

```json
{
  "dependencies": {
    "lodash": "^4.17.21"
  },
  "devDependencies": {
    "vite": "^5.2.0"
  }
}
```

The `^` prefix means "compatible with this version" — npm can install newer minor/patch versions but not a new major version.

### Restoring Dependencies

When you clone a repository, `node_modules/` is not present (it is `.gitignore`d). Restore all dependencies with:

```bash
npm install
# Reads package.json and installs everything listed
```

## The `package-lock.json` File

`package-lock.json` records the exact version of every installed package — including transitive dependencies (packages that your packages depend on).

**Always commit `package-lock.json`.** It ensures everyone on the team (and your CI pipeline) installs the exact same versions. Differences in transitive dependency versions cause "works on my machine" bugs.

> **⚠️ Warning:** Never manually edit `package-lock.json`. It is generated by npm. If it gets corrupted, delete it and `node_modules/`, then run `npm install` to regenerate both.

## npm Scripts

Scripts in `package.json` let you define project commands with short names:

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview",
    "lint": "eslint src/"
  }
}
```

Run them with:

```bash
npm run dev        # start development server
npm run build      # build for production
npm run lint       # run linter
```

Two special scripts run without the `run` keyword: `npm start` and `npm test`.

---

## Uninstalling Packages

```bash
npm uninstall lodash              # remove a dependency
npm uninstall --save-dev eslint   # remove a dev dependency
```

This removes the package from `node_modules/` and from `package.json`.

---

## Common npm Commands Summary

| Command | Effect |
| :--- | :--- |
| `npm init -y` | Create `package.json` with defaults |
| `npm install` | Install all dependencies from `package.json` |
| `npm install <pkg>` | Install and add to `dependencies` |
| `npm install --save-dev <pkg>` | Install and add to `devDependencies` |
| `npm uninstall <pkg>` | Remove a package |
| `npm run <script>` | Run a named script from `package.json` |
| `npm outdated` | List packages with newer versions available |
| `npm update` | Update packages to the newest allowed version |
| `npm audit` | Check for known security vulnerabilities |

## A Minimal Dev Setup with Vite

Vite is the modern front-end build tool — fast dev server, hot module replacement, and a production build system. Set it up in seconds:

```bash
mkdir my-vite-app && cd my-vite-app
npm create vite@latest . -- --template vanilla
npm install
npm run dev
```

Your site is live at `http://localhost:5173` with instant hot reload.

> **💡 Tip:** Vite will be used throughout the rest of this course and is the build tool for the capstone project. Get comfortable with `npm run dev` and `npm run build` now.

## Key Takeaways

- Node.js runs JavaScript outside the browser; npm is its package manager.
- `npm init -y` creates `package.json`, the project manifest.
- `npm install <pkg>` downloads a package and records it in `package.json`.
- `node_modules/` is never committed; `package-lock.json` always is.
- `npm run <script>` executes named commands defined in `package.json`.

## Challenge

Initialize a Vite project and deploy it to GitHub Pages:

1. Run `npm create vite@latest my-vite-site -- --template vanilla`
2. Install dependencies: `cd my-vite-site && npm install`
3. Customize `index.html` — change the title and add some content
4. Add and commit all files (except `node_modules/`; verify `.gitignore` exists)
5. Create a new GitHub repository and push the code
6. Deploy to GitHub Pages — note: for Vite, the build output is in `dist/`; you will need to either push `dist/` or use GitHub Actions

**Success criteria:**
- [ ] `package.json` shows `vite` as a `devDependency`
- [ ] `node_modules/` is in `.gitignore` and NOT committed
- [ ] `package-lock.json` IS committed
- [ ] `npm run dev` starts a dev server at `localhost:5173`
